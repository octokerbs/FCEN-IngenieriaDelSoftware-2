## The need for specification
El tester y el developer tienen que estar "en la misma pagina" con respecto a la especificacion. Ante un conflicto en el pipeline el developer puede argumentar que los tests estan mal hechos con respecto a lo que el implementó mientras que el tester puede argumentar que la implementación está mal modelada con respecto a lo que se quiere testear. 

**El testeo es una forma de chequear la consistencia entre la especificación y la implementación.**

La especificación es "common grounds". El developer escribe la implementación. El tester escribe la especificación. Al ser independientes se reducen las chances de que ambos cometan los mismos errores por separado.

## Observaciones del testeo
Los recursos son finitos. Una cantidad limitada de tests puede ser escrita y corrida por una porción de código. Los tests solo puden **chequear** ciertos aspectos limitados del codigo, no pueden chequear todos los casos posibles. Es como que del universo de tests para un comportamiento del codigo agarramos a un miembro representativo.

La especificacion cambia con el tiempo. Los recursos limitados (tests) tienen que evolucionar con la especificacion con el paso del tiempo. Se explora la idea del test automatizado.

El testeo de caja blanca permite al tester construir un conjunto de tests mejor que el de caja negra y con mejor covertura pues puede ver exactamente que esta pasando.

## Pre y Post conditions
La precondicion es un predicado que se asume verdadero antes de ejecutar una funcion/metodo. La postcondicion es un predicado que se asuume verdadero despues de ejecutar una funcion/metodo cuando tambien la precondicion se asumio verdadera.

Queremos testear casos donde se asumen ciertas verdades del input y queremos verificar que la funcion mantiene la postcondicion luego de que esta fue ejecutada con las verdades del input (precondicion).

Esto nos da un framework de trabajo. Dada una precondicion escribimos un test con un input que la satisfaga. Si el test cumple la postcondicion entonces el test pasa. Si el testo no cumple la postcondicion el test falla. Esto implica que los tests de caja negra son una forma de chequear el estado del programa luego de la ejecucion del metodo.

"Dado que x, y, z entonces deberian suceder r, s, t"
"Si se cumplen x, y, z pero no suceden r, s, t entonces llegamos a un absurdo"
"Si no se cumplen alguno de x, y, z entonces no tiene por que suceder r, s, t"

## Que tan buenos son nuestros tests?
Una forma es code coverage, verificamos que porcentaje de lineas de codigo fueron ejecutadas cuando se corrieron todos los tests del modulo con respecto a todo el source code. O sea, medimos 
$$
\frac{Lineas \ ejecutadas}{Lineas \ totales} \times 100
$$
Tambien existe *branch coverage*, *function coverage* y *statement coverage*. Ojo, *Line coverage* no engloba al resto de coverturas y *function coverage* no engloba a *Line coverage*
$$
Function \ coverage \;\not\!\!\!\implies Line \ coverage
$$
```python
def procesar(x): 
	if x > 0: 
		return "positivo" 
	return "negativo" # esta línea nunca corre
	
# Test: procesar(1)
# Function coverage: 100% → la función fue invocada 
# Line coverage: 75% → la última línea nunca se ejecutó
```
$$
Line \ coverage \;\not\!\!\!\implies Branch \ coverage
$$
```python
def descuento(precio, es_vip): 
	if es_vip: 
		return precio * 0.8 # Si el test solo prueba es_vip=True, 
	return precio           # esta línea nunca se ejecuta
```
$$
Line \ coverage \;\not\!\!\!\implies Statement \ coverage
$$
```python
def check(x): return "positivo" if x > 0 else "negativo"
# Line coverage: 100% → la línea se ejecutó 
# Statement coverage: 50%** → el statement `"negativo"` nunca se evaluó 
```

$$
100 \% \ Branch \ coverage \implies 100 \% \ Statement \ coverage \implies  100\% \ Line \ coverage \implies 100\% \ Function \ coverage
$$
Otra forma es *mutation analysis*. Se asume que desde un inicio el programa esta cerca de ser correcto. Que tiene pocos errores. La idea es testear variaciones (mutantes) del programa. Por ejemplo:

- Reemplazar x > 0 por x < 0
- Reemplazar w por w + 1 o w - 1

Si el conjunto de tests es robusto, deberiamos esperar que algunos de los mutantes fallen los tests mientras que el programa original pase los tests. 

Si descubrimos que un conjunto de mutantes pasan todos los tests implica que nuestro conjunto de tests original no es adecuado y deberiamos agregar tests nuevos que diferencien a nuestro programa original de los mutantes.

```cpp
// Original
int foo(int x, int y) {
	int z = 0;
	if (x <= y) {
		z = x;
	} else {
		z = y;
	}
	return z;
}

// Mutante 1
x <= y -> x > y

// Mutante 2
X <= y -> x != y

// Tests
foo(0, 1) == 0;
foo(0, 0) == 0;

// Resultados
foo(0, 1) == 0; // Pasa
foo(0, 0) == 0; // Pasa

foo1(0,1) == 0; // No pasa
foo1(0,0) == 0; // Pasa

foo2(0,1) == 0; // Pasa
foo2(0,0) == 0; // Pasa

// Conclusion
No es adecuado, el mutante 2 tambien pasa el test suite.

// Resolucion
foo(1, 0) == 0; // Pasa
foo(2, 0) == 0; // Pasa

foo1(1,0) == 0; // No pasa
foo1(2,0) == 0; // No pasa

foo2(1,0) == 0; // No Pasa
foo2(2,0) == 0; // No Pasa
```

## Terminologia
- **ERROR**: Irse por otro lado que nada tiene que ver con la especificacion.
- **DEFECTO**: Un error en el codigo del programa. Hablamos del texto. Puede llevar a una infeccion. (No Siempre). La ejecucion de una linea defectuosa no necesariamente lleva a una infeccion.
- **INFECCION**: Un error en el estado del programa. Un defecto en ejecucion. Puede llevar a una falla (No Siempre). Un error en el estado del programa no necesariamente deriva en una falla. Puede haber un error logico y el programa terminar bien.
- **FALLA**: Un error visible en el comportamiento del programa. 

## Objetivo del Testing
1. Disparar el error -> Cubrir tanto comportamiento como sea posible.
2. Reconocer el error -> Proveer un oraculo.
3. Identificiar funcionalidad faltante -> Tener una especificacion.

## Mutation Analysis - Parte 2
La idea es modificar un programa original para ver como cambian los resultados de los tests. 

Si yo modifique un programa con un mutante y mi test suite no lo detecto (pasaron todos los tests) entonces hay algun tipo de defecto en mi programa. Pues me da el mismo resultado para el codigo original que para uno con logica modificada.

Si el mutante sigue vivo (pasa todos los tests), tenemos que agregar mas tests para tratar de detectarlo.

$$
Mutation \ Score = \frac{Mutantes \ Muertos}{Total \ Mutantes}
$$
El objetivo es que el *Mutation Score* se acerque al 100%.


Para esto se usan operadores de mutaciones. Estos operadores reemplazan valores / operaciones del programa original.

- **Operador de Mutacion ABS**: Reemplazar operaciones unarias. Cambiamos `x` por `abs(x)` o `-abs(x)`
- **Operador de Mutacion AOR**: Reemplazar operaciones binarias. Cambiamos `x + y` por `x * y` o `x - y` o `x` o `y`. O sea, reemplazamos una operacion binaria por un solo simbolo. `x + y` pasa a ser solo `x` o pasa a ser solo `y.
- **Relational Operator Replacement**: Reemplazar predicados. Cambiamos `y != 0` por `y > 0` o por `true` o `false`.

## Tipos de mutantes
- **Mutantes de Primer orden (FOM)**: Exactamente una mutacion. Cada operador puede tener multiples FOM (ABS tiene 2, `abs(x)` y `-abs(x)`)
- **Higher Order Mutant (HOM)**: Mutacion de otra mutacion. 

## Tipos de mutaciones
- **Mutacion Debil**: La mutacion solo afecta el estado del programa (infeccion)
- **Mutacion Fuerte**: La mutacion afecta al output del programa (falla)

La mutacion debil se basa en comparar el estado interno del programa luego de la mutacion debil. Asi nos ahorramos todo el tema de seguir con la ejecucion. O sea, chequeamos el estado despueus de la infeccion para ver como salen los tests, no despues de quue el programa termino.

## Meta Mutantes
En lugar de generar distintaes versiones del mismo programa, generar uuna unica version del programa quue tenga todos los mutantes embebidos. Y yo con una instruccion decido que mutante quuiero ejecutar en este momento.

En vez de compilar muchas veces solo compilo una vez y elijo que mutante quiero ejecutar.
